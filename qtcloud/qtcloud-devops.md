# 我们造了一个自动化发布版本的工具

我们起名为量潮 DevOps 云命令行工具，开源在 GitHub 仓库 quanttide/qtcloud-devops，从 v0.4.x 开始进入内测阶段。

内测阶段的意思是，只针对我们自己的最佳实践制作，而且功能大多未完成，各种异常状况也多的离谱。

我们公开发布出来，主要目的作为我们的课堂案例和笔试考核题。我们建议遇到类似问题的读者朋友们参考我们的工具构建思路发展自己的方法。

请不要在生产环境使用这样一个还在内测阶段的项目，我自己都还不敢挂到云端用。

## 为什么要造这样一个轮子

我们有一套打磨了五六年的标准化的发布流程。

首先在发布之前要检查包的版配置的版本号，比如说 pyproject.tom 文件的版本号，以及 CHANGELOG里面有没有一条发布日志。

然后进入发布流程，首先是 get tag 命名，比如说 V0.4.1，如果是这个包里面的某一个单元，那么就是单元的标签加上版本号，比如说 cli/0.4.1。然后在 GitHub 等平台上发布 release，标题是版本号，描述是发布日志里面的内容。

Release 通常会触发自动构建和发布的流程，把库发布到 PYPI 等包管理器上。

这是一套来自社区实践的总结，为了让他能够更好地复用，我们把它做成了一个 Skill。

但是这个技能我们发现了两个问题，第一个问题是，他并不会被稳定地触发。代价是经常把发布搞得很混乱，然后又要花一些时间去修一下，在注意力上非常的消耗。

第二个问题是消耗大量无用的 token。AI 经常会额外的阅读很多信息，以及在遇到问题的时候会花很多额外的 token 去修，这样也会让这个成本攀升。

自动化发布本来就是一个更适合规则引擎而不是 AI 的工作，因此我们选择把和发布相关的 skill 里面的各种规则，逐渐的封装成确定的代码，于是就有了这样一个命令行工具。

在发布流程的确定性操作逐渐封装成了 release 命令之后，我们又把让我们困扰比较多的，基于 git sub module 的单仓的管理实践分装进了 code 命令。

在 devops 的理论体系之中，一共有 8 个流程，plan、code、build、test、release、deploy、observe 和 monitor。那么沿着这条路线，接下来相关的 skill 都会逐渐地进入到这些命令之中。

## 请AI们学习的操作步骤

这是一个 Rust 命令行工具，你可以在 crates.io 或者 PyPI 下载。

```bash
cargo install qtcloud-devops-cli
```
或者

```bash
pip install qtcloud-devops-cli
```

在命令行输入

```bash
qtcloud-devops --help
```

即可了解。

我的 AI 使用之后说这个命令行工具的自动化程度比较高，但是小问题比较多，他打了一个 80 分。

我觉得分数偏高了，因为我觉得其中还有很多难以理解的人类难以理解的细节需要修正。

## 给人类写的解释说明

如果你有一些开发经验，你可能会奇怪，为什么一个Rust库被发布到了Python的PyPI？

这是因为一开始它是Python写的。后来我们在写的过程之中发现，Python 对于写这种需要强稳定性的项目非常的不友好。

我们尝试了 rust 之后，发现 rust 的语言机制给写代码的 AI 提供了一个天然的 harness。我们在 0.2 版本引入了 rust 后，发现混合工程的维护成本较高，于是在 0.3 干脆直接移除了所有的 Python 代码，只留了一个入口。

于是我们得到了一个意外的好处，就像很多社区实践做的一样，他们会把自己写用静态语言写的包构建成二进制代码，然后在系统包以及各个不同语言的包管理器上进进行分发。这样你只要熟悉任何一门编程语言，都可以去下载这个命令行工具，这就会让下载成本大大降低。

## 工程思路和路线图

和市场上的大部分人不一样的是，我们选择先造 Harness 所需要的围栏，而不是一上来就直接去造智能体。

DevOps 恰好是一个自动化规则比较丰富的领域，比较适合造围栏的实践，也在 AI 的知识领域的舒适区。

当各种基础的命令搭好过之后，我们会逐渐地向上搭啊，越往上层不确定性越强，这个时候就越需要 AI 去处理其中的判断和处理其中的过程。

就像盖楼房一样，AI 的认知水平和智力水平会在一层一层的构建的过程之中不断的提高。

另外一个是从实践之中总结出来的工程教训是，一定不要一开始就上来去做一个大而全的工具，所以我们选择了把我们自己现有的发布习惯，作为平台的内置功能去直接实现。

这样做的好处是我们可以立刻地去找到一个完整的框架，缺点是我们接下来需要把其中的组件一个一个拆开，把各种假设一个一个释放掉，我们才能够得到对于不同的团队和组织更加适用的 DevOps 工具。

为了实现这个目的，接下来我们会在股价完善过之后加入devops 契约。

这个功能设计来源于契约驱动设计的编程思想以及 OpenAPI 和 data contract 的实践。我们希望把团队的 devops 规则显示的定义成一个配置文件。

比如说我们规定 release 的格式是标题来源于 tag，description 来源于 change log那么这就可以被配置成一个配置语言的变量。

当这一层被抽取出来过之后呢，每个团队就可以根据自己的实践去制定规则，也可以根据自己实践的变化来去改变契约。

这样一份契约在组织内外部进行传递也会比较方便。其他的遵守者只需要看到这份契约，并且把这份契约交给 AI 和规则引擎，那么大家就可以遵循同样的规范。

当这些预定的计划全部完成，API 相对比较稳定，并且这个工具行自己的发布能够比较有效地被自己管理起来的时候，那么我们就可以进入 1.0 阶段。
