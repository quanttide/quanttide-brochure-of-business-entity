# 我们造了一个自动化发布版本的工具

我们起名为量潮 DevOps 云命令行工具，开源在 GitHub 仓库 quanttide/qtcloud-devops，从 v0.4.x 开始进入内测阶段。

内测阶段的意思是，只针对我们自己的最佳实践制作，而且功能大多未完成，各种异常状况也多的离谱。

我们公开发布出来，主要目的作为我们的课堂案例和笔试考核题。我们建议遇到类似问题的读者朋友们参考我们的工具构建思路发展自己的方法。

请不要在生产环境使用这样一个还在内测阶段的项目，我自己都还不敢挂到云端用。

## 为什么要造这样一个轮子

我们有一套打磨了五六年的标准化的发布流程。

首先在发布之前要检查包的版配置的版本号，比如说 pyproject.tom 文件的版本号，以及 CHANGELOG里面有没有一条发布日志。

然后进入发布流程，首先是 get tag 命名，比如说 V0.4.1，如果是这个包里面的某一个单元，那么就是单元的标签加上版本号，比如说 cli/0.4.1。然后在 GitHub 等平台上发布 release，标题是版本号，描述是发布日志里面的内容。

Release 通常会触发自动构建和发布的流程，把库发布到 PYPI 等包管理器上。

这是一套来自社区实践的总结，为了让他能够更好地复用，我们把它做成了一个 Skill。

但是这个技能我们发现了两个问题，第一个问题是，他并不会被稳定地触发。代价是经常把发布搞得很混乱，然后又要花一些时间去修一下，在注意力上非常的消耗。

第二个问题是消耗大量无用的 token。AI 经常会额外的阅读很多信息，以及在遇到问题的时候会花很多额外的 token 去修，这样也会让这个成本攀升。

自动化发布本来就是一个更适合规则引擎而不是 AI 的工作，因此我们选择把和发布相关的 skill 里面的各种规则，逐渐的封装成确定的代码，于是就有了这样一个命令行工具。

在发布流程的确定性操作逐渐封装成了 release 命令之后，我们又把让我们困扰比较多的，基于 git sub module 的单仓的管理实践分装进了 code 命令。

在 devops 的理论体系之中，一共有 8 个流程，plan、code、build、test、release、deploy、observe 和 monitor。那么沿着这条路线，接下来相关的 skill 都会逐渐地进入到这些命令之中。

## 请AI们学习的操作步骤

这是一个 Rust 命令行工具，你可以在 crates.io 或者 PyPI 下载。

```bash
cargo install qtcloud-devops-cli
```
或者

```bash
pip install qtcloud-devops-cli
```

在命令行输入

```bash
qtcloud-devops --help
```

即可了解。
